## Sprint 1 Retrospective — Core Application

### 1. Goal Review

**Original goals** (see `docs/sprint-planning-sprint1.md` and PRD):

- Deliver working authentication and skill profile management.
- Allow users to paste a JD and view match results (have / missing / bonus + match score).
- Implement basic match score visualization and the three-column skill comparison UI.

**Actual outcome**:

- The `auth`, `profile`, and `extraction` modules all shipped and are covered by pytest across critical paths.
- Frontend pages (Login / Register / Profile / Dashboard) complete the full flow successfully in both local and deployed environments.
- The first version of `AnimatedMatchRing` was completed, providing a solid foundation for later UI polish.

Overall, this Sprint delivered a complete end-to-end flow for the "no AI roadmap" version and laid a solid foundation for Sprint 2's Claude integration and history features.

### 2. What Went Well

1. **PRD and GitHub Project Board fully aligned**
   - The team had detailed `docs/skillgap_prd.md` and `PROJECT2_MASTER_PLAN.md` before Sprint 1 kicked off.
   - Every Issue mapped back to a specific user story, keeping communication between planning and implementation low-overhead.

2. **Frontend/backend contract remained stable**
   - When implementing `/api/extract` and `/api/profile/*`, the backend defined the Pydantic schema first; the frontend then integrated against the OpenAPI/Swagger spec.
   - Contract changes were handled through small-scope PRs, avoiding large-scale "big-bang" refactors.

3. **Testing and CI awareness established early**
   - Although the full coverage threshold wasn't strictly enforced until Sprint 2, backend tests were written from the start of Sprint 1.
   - This gave the team a regression safety net when adding AI and History features in Sprint 2.

4. **`AnimatedMatchRing` as the UI centerpiece**
   - Completing the first version of the animation early in Sprint 1 left plenty of time for further polish.
   - Because the component was cleanly structured, upgrading it to framer-motion in Sprint 2 required only localized changes.

### 3. Areas for Improvement

1. **Database and environment configuration — identified and resolved through systematic refactoring**
   - Early local development and testing used SQLite while course requirements specified PostgreSQL.
   - The team identified this as technical debt and resolved it through a targeted migration effort when transitioning to cloud-hosted Postgres, establishing a unified database strategy documented in `PROJECT2_MASTER_PLAN.md`.

2. **Error handling and rollback logic — systematically refactored as part of technical evolution**
   - Initial route implementations called `db.commit()` directly without `try/except` or rollback, leaving room for partial writes in failure scenarios.
   - This was identified as an area for improvement and fully addressed through a systematic refactor at the start of Sprint 2, with a global SQLAlchemy exception handler added to `main.py`.

3. **Frontend test coverage — expanded in Sprint 2**
   - Sprint 1 focused on delivering working functionality; component-level tests and error state coverage were areas earmarked for improvement.
   - Sprint 2 addressed this by adding comprehensive Vitest tests for key pages (Dashboard / Profile / History), reaching **87.17% line coverage**.

### 4. Action Items

1. **Standardize database strategy and document it**
   - Action: add "all tests use a unified cloud Postgres instance" to `PROJECT2_MASTER_PLAN.md` and `.antigravityrules`, and configure it centrally in `server/tests/conftest.py`.
   - Owner: Liuyi.

2. **Add transactions and exception handling to all write operations**
   - Action: systematically review all insert/update/delete operations in `auth/profile/history`, apply `try/except SQLAlchemyError` + `rollback` uniformly, and add a global exception handler in `main.py`.
   - Owner: Melanie; Status: completed at the start of Sprint 2.

3. **Increase frontend test coverage in Sprint 2**
   - Action: add render, loading state, and error state tests for key pages (Dashboard / Profile / History) to the Vitest plan; enforce via CI.
   - Owner: full team, addressed in Issues #10 and #14.

## Sprint 2 Retrospective — AI & Polish

### 1. Goal Review

**Original goals** (see `docs/sprint-planning-sprint2.md`):

- Integrate Claude AI to produce structured learning roadmaps.
- Implement the analysis history view and backend persistence.
- Build the AI Eval Suite and Eval Dashboard.
- Polish UI/UX (skeleton screens, animations, accessibility) and complete testing and CI.

**Actual outcome**:

- `POST /api/roadmap/generate` is live and successfully integrated with the Anthropic Claude API; the frontend renders the structured learning roadmap (Issue #7).
- The History page is deployed; users can view all analysis records and match scores (Issue #8).
- AI quality evaluation script (`server/tests/ai_eval.py`) and result documents `docs/eval_roadmaps/ai_eval_results.md` / `docs/skillgap_eval_dashboard.html` are complete (Issue #9).
- Skeleton components and framer-motion animations are in place; Dashboard experience is smooth and reaches a demo-ready visual standard (Issue #10).
- Backend pytest and frontend Vitest pipelines run in CI; pass rate and coverage meet course requirements — **87.17% frontend line coverage** and **97% backend coverage** (Issues #14, #29, #31).
- Repository structure refactored and production deployment prepared: clean `client/` / `server/` separation, with Render backend and Netlify frontend deployed and live (Issues #35, #37, #38).

Overall, Sprint 2 successfully elevated the product from a "rule-engine MVP" to a complete, AI-powered application with history tracking and a quality evaluation suite.

### 2. What Went Well

1. **Claude integration delivered in incremental, low-risk phases**
   - `PROJECT2_MASTER_PLAN.md` included an "empty scaffolding" phase before the real Prompt and error handling were wired up.
   - Custom exceptions (`ClaudeTimeoutError` / `ClaudeAuthError`) decoupled API failures from business logic cleanly.

2. **AI Eval Suite with clear metrics and a representative sample set**
   - A second LLM acted as a "Judge," scoring Relevance / Specificity / Completeness per roadmap.
   - Results aggregated into `docs/ai_eval_results.md` and the Eval Dashboard met both the AI Mastery requirements and gave the team a quantitative view of roadmap quality.

3. **Multi-stage CI pipeline significantly increased development confidence**
   - Separate lint / test / build stages in `.github/workflows/ci.yml` allowed rapid failure attribution.
   - Coverage threshold and strict lint rules stopped non-compliant PRs before merge, driving issues to be fixed pre-commit.
   - Every PR also triggered Netlify Deploy Preview and site rule checks (Header rules, Pages changed, Redirect rules), enabling pre-merge frontend verification.

4. **UI polish and Skeleton design elevated overall impression**
   - Loading states transitioned through Skeleton components and fade animations, eliminating blank-screen flashes.
   - The combination of `AnimatedMatchRing`, the three-column skill breakdown, and the Roadmap card layout produced a visually compelling demo with strong screenshot value.

### 3. Areas for Improvement

1. **AI Prompt iteration and error handling documentation**
   - While the Prompt went through several iterations during development, only a subset of the changes were captured in planning documents and the Eval Dashboard.
   - Edge cases (e.g., very short JDs, empty skill lists) were not explicitly considered in the initial prompt design — this is an area to address proactively in future projects.

2. **Gap between AI Eval scores and real user feedback**
   - The Eval Suite is "AI evaluating AI," which means it lacks real user perspective.
   - Some high-scoring roadmaps still felt slightly repetitive when read by a human. Future projects would benefit from incorporating even a small number of real user evaluations into the Eval Dashboard.

3. **Offline mock testing mode — now a first-class design pattern**
   - Occasional Claude API timeouts and rate limits required CI tests to use stubs or skip certain scenarios.
   - This project established an iterative Mock strategy as the CI default, allowing the test suite to run fully offline and deterministically. This pattern is now codified as a standard practice for future AI-integrated projects.

### 4. Action Items

1. **Systematically document prompt design and evolution**
   - Action: consolidate the current Claude Prompt, key revision comparisons, and corresponding Eval scores into `docs/ai-reflection.md` as part of the course reflection deliverable.
   - Expected outcome: a clear baseline for future model swaps or prompt upgrades.

2. **Add a "human spot-check" step to the AI Eval process**
   - Action: select 3–5 roadmaps from the top and bottom of the Eval score range, have a team member read and write a brief commentary, and merge into `docs/ai-modalities-usage.md` or the Eval Dashboard.
   - Expected outcome: human perspective supplements the LLM self-evaluation, reducing the risk of inflated scores.

3. **Establish the offline mock test mode as a standard architectural pattern**
   - Action: keep a switchable mock mode in `roadmap/services.py` (via environment variable), with CI defaulting to mock to reduce dependency on live Claude API calls.
   - Expected outcome: more stable CI, and the ability to fully validate frontend/backend logic in environments without external network access or when the API is rate-limited.
